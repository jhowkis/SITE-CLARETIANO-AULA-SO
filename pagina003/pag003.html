Rastreamento de Memória Livre (O "Mapa" do SO)

Na multiprogramação com partições variáveis, a memória torna-se um mosaico de processos ativos e "buracos" (espaços vazios) intercalados. Para que o Sistema Operativo consiga alocar um novo programa de forma eficiente, ele deve monitorizar constantemente a localização e o tamanho destes espaços. Para isso, utilizam-se duas estruturas de dados principais:

  1. Mapa de Bits "Bitmaps": Nesta abordagem, a memória é dividida em pequenas unidades de alocação (que podem variar de alguns bytes a kilobytes). O SO mantém um registo binário onde cada unidade é representada por um único bit:
    • 0: Indica que a unidade está Livre (Buraco).
    • 1: Indica que a unidade está Ocupada (Processo).
      • Vantagem: É uma estrutura extremamente simples e de tamanho fixo.
      • Desvantagem: A busca por um espaço contíguo grande exige que o SO percorra a tabela à procura de uma sequência longa de zeros, o que se torna computacionalmente lento em memórias extensas.
      • Mapa de Bits: 1111 00 111 0000000
  2. Listas Encadeadas (Linked Lists):Considerado um método mais sofisticado e eficiente para memórias dinâmicas, o SO mantém uma lista de "nós" (elos) conectados. Cada nó descreve um segmento contínuo da memória, contendo obrigatoriamente três informações:
    • Tipo: Se o segmento é um Processo (P) ou um Buraco (H - Hole).
    • Endereço: Onde o bloco começa na RAM.
    • Comprimento: O tamanho total do bloco.
      • Vantagem: A atualização é muito mais rápida. Quando um processo termina, o SO apenas altera o tipo do nó de P para H e, se houver buracos vizinhos, funde-os num único nó grande. Além disso, facilita a execução de algoritmos como First-Fit ou Best-Fit.
      • Exemplo Prático de Lista Encadeada: Usando o mesmo cenário acima (Processo A de 4 unidades no início, seguido de um buraco de 2 unidades e o Processo B de 3 unidades): (P, 0, 4) ➔ (H, 4, 2) ➔ (P, 6, 3) ➔ (H, 9, 7)

Estratégias de Posicionamento de Memória

Na multiprogramação, após mapear os espaços livres na RAM (usando Bitmaps ou Listas Encadeadas), o Sistema Operativo enfrenta um novo desafio: quando um novo programa é aberto, em qual dos vários "buracos" vazios ele deve ser colocado? Para tomar esta decisão de forma automatizada, o SO utiliza algoritmos de alocação.

Nota Técnica: O que é um Algoritmo neste contexto? Um algoritmo é uma sequência finita de passos lógicos ou regras programadas no núcleo do Sistema Operativo. O seu objetivo aqui é varrer a estrutura de dados que mapeia a memória (como a lista encadeada) e calcular, com base no tamanho do processo e nos espaços disponíveis, o local exato onde o programa será carregado.

Existem três algoritmos clássicos para realizar este posicionamento:

  • First-Fit (Primeiro Encaixe): O algoritmo varre a lista de espaços livres desde o início e aloca o programa no primeiro buraco que for suficientemente grande.
    • Característica: É o método mais rápido e simples, pois a busca é interrompida assim que um espaço válido é encontrado, não sendo necessário ler a lista inteira.
  • Best-Fit (Melhor Encaixe): Este algoritmo procura o buraco que tenha o tamanho mais exato possível (o menor espaço onde o programa consiga caber).
    • Característica: A sua intenção é não desperdiçar grandes espaços vazios. No entanto, é mais lento (exige a leitura de toda a lista) e, paradoxalmente, gera o pior tipo de fragmentação externa: ao deixar sobras muito pequenas, cria dezenas de "mini-buracos" inúteis onde nenhum outro processo conseguirá entrar.
  • Worst-Fit (Pior Encaixe): Funciona de forma diametralmente oposta ao Best-Fit. Ele varre toda a memória para encontrar deliberadamente o maior buraco disponível e coloca lá o programa.
    • Característica: A ideia por trás desta estratégia é que a "sobra" de memória após a alocação ainda seja um bloco grande o suficiente para acomodar outro programa útil, tentando evitar a criação dos "mini-buracos" do Best-Fit.
  • Afinal, qual é o melhor algoritmo?
    • Apesar do nome sugestivo, o First-Fit é considerado o melhor algoritmo na prática. Simulações clássicas de Sistemas Operativos (como demonstrado por Tanenbaum) provam que o First-Fit ganha por ser muito mais rápido de executar e por manter uma taxa de aproveitamento de memória incrivelmente boa e equilibrada. O Best-Fit perde por ser lento e fragmentar demasiado a memória a longo prazo, enquanto o Worst-Fit apresenta quase sempre o pior desempenho global nos testes reais.
